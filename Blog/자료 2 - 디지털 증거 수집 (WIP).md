## 1. 디지털 증거 특성
디지털 증거는 중요한 특성 몇 가지가 있다. 

가장 먼저 저번 글에서 말했다시피 디지털 증거는 휘발성인 경우가 많다. 휘발성의 의미를 검색해보면 "액체가 기체로 변하여 쉽게 날아가 흩어지는 성질" 이라고 나오는데 휘발성 증거는 이처럼 쉽게 지워지는 데이터를 말한다. 이런 특성 때문에 디지털 증거는 신속하게 수집하는 것이 중요하다. 

또한 디지털 증거는 변조가 쉽다는 특성이 있다. 증거가 결국 파일 형태로 저장이 되기 때문에 누구나 마음 먹으면 쉽게 삭제, 추가, 심지어는 변형까지 가능하다. 그러므로 수집했을 때와 법정에 제출했을 때 동일한 데이터라는 것을 증명하는 무결성이 중요하다. 

이러한 특성 때문에 조금만 실수해도 증거자료가 법적 효력이 없어지고 휘발성 증거의 경우 다시 수집하는 것 또한 불가능하기 때문에 디지털 증거를 수집하는 과정을 숙지하고 있는 것은 필수적이다. 

그렇다면 디지털 증거를 어떻게 수집해야 할까? 먼저 상대적으로 간단한 비휘발성 증거 수집에 대해 알아보겠다. 

## 2. 비휘발성 증거 수집 과정
비휘발성 증거 수집은 쉽게 말하면 하드 디스크나 SSD 같은 보조 저장 장치의 모든 데이터를 추출하여 하나의 파일(이미지)에 저장하는 과정이다. 간단해 보이지만 그래도 반드시 알아야 하는 내용이 몇 가지 있는데 이에 대해 설명 하겠다.

#### 2.1. 디스크 access 방식
디스크(SSD도 해당 되지만 편의상 디스크라고 하겠다)를 access하는 방식은 크게 직접 access하는 direct access와 운영체제가 BIOS를 통해 access하는 BIOS access 두 가지가 있다. 

BIOS access는 디스크의 하드웨어 정보를 몰라도 가능하기 때문에 편리하다는 장점이 있지만 BIOS를 통하기 때문에 BIOS의 설정이나 디스크 포맷 방식에 따라 디스크보다 작은, 불완전한 이미지가 생성될 수 있다는 단점이 있다. 

반면에 direct access는 디스크를 직접 access하기 때문에 디스크의 숨겨진 데이터(HPA, DCO, slack space)까지도 완벽하게 복제할 수 있고 속도도 더 빠르지만 디스크와 소통할 수 있는 드라이버가 있어야 한다는 단점이 있다. 해당 드라이버만 있다면 direct access로 이미지를 생성하는 툴을 사용하는 것이 당연히 유리하다.

#### 2.2. 증거 수집 방식
비휘발성 증거 수집에는 크게 두 가지 방법이 있다.

첫 번째는 suspect(의심스러운) 운영 체제를 통해서 증거를 수집하는 live acquisition이다. 운영 체제가 어떻게 변형됐을지 모르기 때문에 live acquisition은 가능하면 피하는 것이 좋다. 

두 번째는 suspect 운영 체제의 도움 없이 증거를 수집하는 dead acquisition이다. 신뢰하는 운영 체제를 통해 증거를 수집하기 때문에 데이터가 변형 됐을 가능성은 극히 적다. 용의자가 하드웨어 레벨에서 데이터를 막도록 변형했을 가능성도 있지만 이는 흔하지 않으며 아예 다른 하드웨어를 사용함으로써 피할 수 있다. 

dead acquisition을 기본으로 하고 불가능한 상황에서만 live acquisition을 하는 것을 권장한다. 

#### 2.3. 에러 처리
이미징하는 과정에서 access가 안되는 섹터에 도달하면 에러가 발생하는데 이미징 툴은 이를 올바르게 처리(handle)할 수 있어야 한다. 대체로 이런 에러는 섹터를 기록하고 그 자리에는 0을 채워 넣는 방식으로 처리한다.

#### 2.4. 쓰기 차단 장치
당연하게 디스크를 확보한 후 디스크의 내용이 변경이 되면 안된다. 그러나 프로그램의 오류든 사람의 실수든 디스크의 내용이 변경되는 경우가 있을 수 있다. 특히나 운영 체제에서 디스크를 자동으로 마운트하면 디스크 데이터가 바뀌는 곤란한 상황이 생길 수 있는데 이런 상황을 방지하기 위한 장치가 write blocker(쓰기 차단 장치)이다. 

write blocker은 하드웨어로 구현할 수도 있고 소프트웨어로 구현할 수도 있으며 툴마다 차이도 있기 때문에 사용할 툴이 어떻게 동작하는지 등을 반드시 숙지하고 사용해야 한다. 

#### 2.5. 출력 형식
이미징 툴마다 출력 format(형식)이 다를 수 있다. 복사한 데이터만 있는 raw image format일 수도 있고 각종 메타데이터가 같이 있는 embedded image format일 수도 있다. embedded image format의 경우 호환성이 떨어지기 때문에 이미지 툴의 출력 format을 알아두는 것이 중요하다. 

#### 2.6. 해시
이미지를 만든 후 디스크의 해시값을 계산하여 이미지의 해시값과 비교하여 데이터가 오류 없이 복사 됐는지 확인하는 과정은 필수다. 이 과정을 거치지 않으면 디지털 포렌식의 5대 원칙 중 무결성의 원칙에 어긋나므로 법적 효력을 상실할 수 있다. 

데이터 전체에 대해 해시를 계산하는 대신 데이터를 쪼개서 따로 해시값을 계산하면 오류가 어디서 발생했는지 알 수 있으며 모든 데이터를 다시 복사할 필요가 없어진다. 

이미징 툴의 버그로 인해 같은 데이터를 복사할 때와 해시를 계산할 때 같은 오류가 발생하지 않도록 가능하면 이미징 툴과 독립적인 다른 툴을 사용해서 해시 값을 계산하는 것을 권장한다. 

## 3. 비휘발성 증거 수집 실습
이번에는 공부한 내용을 바탕으로 매우 간단한 비휘발성 증거 수집 실습을 해보겠다.

사용할 장치는 4GB USB이다. 

먼저 USB 장치를 연결하고 *dd*를 사용하여 이미지를 생성했다.
``` data
$ sudo dd if=/dev/sdc of=usb.dd bs=512 status=progress
4022963200 bytes (4.0 GB, 3.7 GiB) copied, 158 s, 25.5 MB/s
7897088+0 records in
7897088+0 records out
4043309056 bytes (4.0 GB, 3.8 GiB) copied, 158.231 s, 25.5 MB/s
```

다음으로 USB 장치의 sha256 해시값과 생성된 image의 해시값을 비교해봤다.
``` data
$ sha256sum /dev/sdc usb.dd
b23e76cedac3c92394653ed3bb688a21225f87564e3231d145bc9f633ed29da2  /dev/sdc
b23e76cedac3c92394653ed3bb688a21225f87564e3231d145bc9f633ed29da2  usb.dd
```

두 해시값이 동일한 것을 볼 수 있다. 

#### *주의* 
말했다시피 이는 매우 간단한 이미징 실습이다. 실제 상황에서는 *umount*를 통해 언마운트를 하고 *blockdev*로 쓰기도 차단한 후 이미지 생성 및 해시값 계산을 하는 것이 안전하다. 또한 망가진 섹터(bad sector)에 도달했을 경우 sha256이 작동을 안할 수 있으므로 에러 처리도 하는 것이 중요하다. 

HPA나 에러 처리 등까지 고려한 자세한 과정을 보고 싶다면 File System Forensic Analysis의 Chapter 3를 보는 것을 추천한다.
