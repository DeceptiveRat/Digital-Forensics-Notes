## 1. kernel modules
- kernel memory overview:
	![[kernel_memory_overview.png]]
- kernel debugger data block has `PsLoadedModuleList` that points to list of `KLDR_DATA_TABLE_ENTRY` structures
- `KLDR_DATA_TABLE_ENTRY` structures contain metadata about each kernel module:
	- base address
	- size of module
	- full path to file on disk
- `modscan` plugin finds unlinked entries via pool-scanning for `MmLd` tags
- for entries zeroed out as well as unlinked, module data has to be found for analysis:
	- brute force scan for PE headers with base address not represented in module list 

## 2. finding suspicious modules
- hidden/unlinked: no legitimate reason to unlink module
- handles interrupts:
	- NT module (*ntoskrnl.exe*, *ntkrnlpa.exe*, etc) should always be the handler of critical interrupts such as page fault, breakpoint, and system service dispatcher
- provides system APIs:
	- *System Service Dispatch Table(SSDT)*, used to resolve address of system APIs in kernel memory, should only involve NT module, GUI subsystem module (*win32k.sys*), IIS support driver (*spud.sys*), and some antivirus products
- unsigned:
	- all kernel modules need to be signed
	- check signer, certificate expiration, etc
	- kernel module files from disk are required to verify signature due to changes when loading modules into memory
- invalid name/path:
	- certain malware use fixed names
	- modules should not be loaded from temporary paths
- installs callbacks:
	- callbacks, which notify when certain events occur, can be used maliciously
- creates devices:
	- device names can be checked for IoC
	- check whether driver's devices are attached to the keyboard, network, or FS driver stacks
- contains known signatures:
	- brute force code content checks for known signatures can be useful

## 3. module load methods
- *Service Control Manager(SCM)*:
	- method: creates service of type `SERVICE_KERNEL_DRIVER` and starts service
	- artifacts:
		- registry subkey under `CurrentControlSet\services`
		- event log messages if auditing is enabled
		- new service record structure in memory of *services.exe*
- `NtLoadDriver`:
	- method: directly calls `NtLoadDriver` instead of `CreateService` and `StartService`
	- artifacts:
		- registry subkey
- `NtSetSystemInformation`: 
	- method: call `NtSetSystemInformation` with `SystemLoadAndCallImage` class
	- does not require registry entries
	- no easy way to unload, besides rebooting machine

## 4. modules on live system
- *process explorer*:
	![[process_explorer_modules.png]]
- Windows API:
	- `EnumDeviceDrivers` can retrieve load address for each kernel module
	- helper APIs call `NtQuerySystemInformation` internally
- *Windows Management Instrument(WMI)*:
	- can use `Win32_SystemDriver` class to enumerate system drivers
	- actually consults reigstry, not `NtQuerySystemInformation`
- *Nirsoft*:
	- shows list of loaded modules generated by calling `NtQuerySystemInformation`
- `NtQuerySystemInformation`:
	- C/C++ programs can directly call `NtQuerySystemInformation` with `SystemModuleInformation` class to retrieve list of kernel modules
	- references doubly linked list of `KLDR_DATA_TABLE_ENTRY` structure
- unlinking metadata structure hides module from most tools
- deleting registry keys or loading module via `NtSetSystemInformation` hides module from WMI

## 5. `modules` plugin
- walks doubly linked list in `PsLoadedModuleList`
- shows order of loaded modules
- example output:
	``` sh
	$ python vol.py -f memory.vmem --profile=Win7SP1x64 modules
	Volatility Foundation Volatility Framework 2.4
	Offset(V) Name Base Size File
	------------------ --------------- ------------------ --------- ----
	0xfffffa8000c32890 ntoskrnl.exe 0xfffff80002852000 0x5ea000 \SystemRoot\system32\ntoskrnl.exe
	0xfffffa8000c327a0 hal.dll 0xfffff80002809000 0x49000 \SystemRoot\system32\hal.dll
	0xfffffa8000c326c0 kdcom.dll 0xfffff80000b9a000 0xa000 \SystemRoot\system32\kdcom.dll
	0xfffffa8000c2cf20 mcupdate.dll 0xfffff88000cdd000 0x4f000 \SystemRoot\system32\mcupdate_GenuineIntel.dll
	[snip]
	```
	- `Offset(V)` is the VA of `KLDR_DATA_TABLE_ENTRY` metadata structure 
	- `Base` is the VA of start of module's PE header

## 6. `modscan` plugin
- uses pool tag scanning method for `MmLd`
- can find unlinked or unloaded modules
- physical offset is displayed instead of virtual offset
- modules appear in order they are found, not loaded
- example output:
	``` sh
	$ python vol.py -f memory.vmem --profile=Win7SP1x64 modscan
	Volatility Foundation Volatility Framework 2.4
	Offset(P) Name Base Size File
	------------------ --------------- ------------------ --------- ----
	0x000000000038ae90 mouclass.sys 0xfffff88003bd9000 0xf000 \SystemRoot\system32\DRIVERS\mouclass.sys
	0x000000002c78c590 serenum.sys 0xfffff88003a1d000 0xc000 \SystemRoot\system32\DRIVERS\serenum.sys
	```

## 7. `unloadedmodules` plugin
- uses kernel list of recently unloaded modules
- unloaded module list conatins timestamps and locations in kernel memory they used to occupy
- useful when rootkits attempt to unload quickly
- example output:
	``` sh
	$ python vol.py -f rustock-c.vmem --profile=WinXPSP3x86 modules | grep xxx
	$ python vol.py -f rustock-c.vmem --profile=WinXPSP3x86 modscan | grep xxx
	$ python vol.py -f rustock-c.vmem --profile=WinXPSP3x86 unloadedmodules
	Volatility Foundation Volatility Framework 2.4
	Name StartAddress EndAddress Time
	-------------------- ------------ ---------- ----
	[snip]
	kmixer.sys 0x00f75ae000 0xf75d8000 2010-12-31 18:46:46
	xxx.sys 0x00f6f88000 0xf6fc2000 2010-12-31 18:47:57
	```

## 8. `moddump` plugin
- extracts kernel modules
- can only extract currently loaded modules with valid PE headers
- example output:
	``` sh
	$ python vol.py –f memory.dmp --profile=Win7SP1x64 moddump --base=0xfffff88003800000 --dump-dir=OUTDIR/
	Volatility Foundation Volatility Framework 2.4
	Module Base Module Name Result
	------------------ -------------------- ------
	0xfffff88003800000 UNKNOWN OK: driver.fffff88003800000.sys
	```
	- for analysis with IDA, `ImageBase` address in PE header must be changed to load address in kernel memory:
		``` python
		$ python
		Python 2.7.6 (v2.7.6:3a1db0d2747e, Nov 10 2013, 00:42:54)
		[GCC 4.2.1 (Apple Inc. build 5666) (dot 3)] on darwin
		Type "help", "copyright", "credits" or "license" for more information.
		>>> import pefile
		>>> pe = pefile.PE("driver.0xfffff88003800000.sys", fast_load = True)
		>>> pe.OPTIONAL_HEADER.ImageBase = 0xfffff88003800000
		>>> pe.write("driver.0xfffff88003800000.sys")
		>>> quit()
		```
		- gives IDA context to properly display relative function calls, jumps, and string references
		- `impscan` plugin may have to be used to generate labels for IDA depending on the state of the import address table
